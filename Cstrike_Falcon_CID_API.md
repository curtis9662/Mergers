## 🦅 1. CrowdStrike Falcon API: CID Scoping

Ingesting CrowdStrike data requires a strict understanding of Customer ID (CID) boundaries. Your approach depends on your environment's tenancy structure.

### Standard Environments (Single Tenant)
In a standard Falcon setup, API clients are generated directly within a specific tenant's console. 

> [!NOTE]  
> **Implicit Scoping:** Any OAuth2 token generated with standard Client ID and Secret credentials is automatically and exclusively scoped to that specific CID. By default, any data you query or ingest is natively restricted to that single tenant.

### Flight Control / MSSP Environments (Multi-Tenant)
If you operate in a multi-tenant environment where a **"Parent" CID** manages multiple **"Child" CIDs**, you can easily restrict your data ingestion to one specific child using your Parent credentials.

To do this, you scope your API token during the initial authentication phase using the `member_cid` parameter.



**Authentication Flow:**
1. Make a `POST` request to the CrowdStrike OAuth2 endpoint.
2. Provide your `client_id` and `client_secret`.
3. Pass the **optional** parameter in the body: `member_cid`.

```http
POST https://api.<cloud>[.crowdstrike.com/oauth2/token](https://.crowdstrike.com/oauth2/token)
Content-Type: application/x-www-form-urlencoded

client_id=YOUR_PARENT_CLIENT_ID
&client_secret=YOUR_PARENT_SECRET
&member_cid=TARGET_CHILD_CID_STRING
[!IMPORTANT]

By passing the exact child CID string into the member_cid parameter, the API gateway returns an access token that is strictly locked to that specific child. All subsequent data ingestion queries will only return data for that specific CID.

🌐 2. CloudConnexa API Integration
The CloudConnexa API allows you to automate the management of your networks, hosts, users, and other Wide-area Private Cloud (WPC) resources.

Integration Checklist
🔑 1. Generate API Credentials

Navigate to API & Logs > API in the CloudConnexa Admin Portal.

Click Create Credentials to receive your Public API Client ID and Secret.

⚙️ 2. Enable API Access

Ensure the Enable API toggle is strictly active in the portal.

🔐 3. Authenticate

Use the client credentials to request an OAuth 2.0 access token.

Endpoint format: https://[company_name].api.openvpn.com

📚 4. Access Documentation

Full API functionality is documented via Swagger, easily accessible from within your Administration Portal.

🏗️ 5. Infrastructure as Code (Automation)

Beyond direct API calls, you can leverage the official CloudConnexa Terraform Provider for seamless infrastructure automation.

📊 3. Devo API Integration
Devo provides a robust HTTP API endpoint designed to ingest data, such as logs, events, or monitoring metrics generated by environments like CloudConnexa.

Steps to Ingest Data
Create a Devo Authentication Token:

In your Devo domain, navigate to Administration → Credentials → Authentication tokens.

Click Create token and define the target table(s) for your events (e.g., my.app.**).

Send Data via HTTP:

Write a script or use an HTTP client to push data from your systems to the Devo API endpoint.

Pass the created token in the headers for authorization.

Bash
# Example curl command pushing a JSON event to Devo
curl -X POST "https://<devo_endpoint>/http" \
  -H "Authorization: Bearer YOUR_DEVO_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"event_data": "Sample CloudConnexa status change"}'
Log Streaming (CloudConnexa ➡️ Devo):

CloudConnexa offers log streaming for status changes and high usage notifications.

Architecture Note: While a direct, out-of-the-box integration isn't specified, these streams can be captured and pushed to Devo using the HTTP API methodology outlined above.
